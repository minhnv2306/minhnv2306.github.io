[Lab] Section 7: Elastic Load Balancing & Auto Scaling Groups

Cùng thực hành một số bài liên quan đến ELB và ASG nhé!

Section 07

Lab 01: Load Balancer (ALB) Hands On

Bước 1: Launch 1 instance EC2 (Amazon Linux 2)

(sử dụng User data)

Bước 2: Launch thêm 1 chiếc EC2 Instance y hệt cái đầu

Bạn chuột phải vào Instance => Launch more like this

Note: Bạn cần change Network tại Step 3: Configure Instance Details

  • Network: Giống Instance 1
  • Subnet: Region khác Instance 1 (để chắc chắn tính khả dụng qua 2 availability zone)

Các điều kiện còn lại giống nhau

Hiện tại chúng ta đã có 2 instance ở 2 AZ khác nhau. Bạn có thể truy cập IP trên trình duyệt để kiểm tra chúng

Bây giờ chúng ta sẽ config cân bằng tải

Bước 3: Config Load Balancing

Từ sidebar bên trái => LOAD BALANCING => Load Balancers => Create Load Balancers

Như phần lý thuyết đã đề cập, chúng ta có 1 vài kiểu LB, trong bài thực hành sẽ sử dụng layer 7 - HTTP

Chọn Application Load Balancer (HTTP/HTTPs)

Step 1: Configure Load Balancer

  • Name: DemoALB
  • Schema: internet-facing
  • IP address type: ipv4

  • Listeners: HTTP => Port 80

  • Availability Zones: chọn tất cả zones để load balancer có tính khả dụng cao

  • Click Next

Step 2: Configure Security Settings

Nếu chỉ HTTP thì bạn bỏ qua bước này, chọn Next

Step 3: Configure Security Group

Create new security group

  • Type: Custom TCP
  • Protocal: TCP
  • Port Range: 80
  • Source: Custom 0.0.0.0/0,””/0 (all IP)

Click Next

Step 4: Configure Routing (Cấu hình kể instance nào chịu tải)

  • Target group: New target group
  • Name: demo-tg
  • Target type: Instance (sẽ chịu tải vào EC2)
  • Protocol: HTTP
  • Port: 80

Health checks (check hoạt động EC2, nếu EC2 oẳng, nó ko gửi request đến nữa)

  • Protocal: HTTP
  • Path: /

Click Next

Step 5: Register Targets

  • Chọn 2 instance ở dưới và Cick Add to registered

Kết quả các instance sẽ xuất hiện trên Registered targets là OK

Click Next, Review và Tạo ALB. Trạng thái sẽ là provisioning, và chờ 1 khoảng thời gian sẽ chuyển về state active. Lúc này bạn sẽ thấy DNS name, chính là domain bạn truy cập ALB

Mở trình duyệt và truy cập vào domain này, bạn sẽ thấy chuyển qua lại giữa 2 Ip của 2 instance. Nếu như vậy là bạn đã thành công!

Bạn có thể xem lại các Config, Target Group để hiểu hơn, cũng như check lỗi

Bạn có thể tuỳ biến Thêm (Register targets) vào Target Groups để tăng số lượng Instance

Bước 4: Kiểm tra Health Check

Như lý thuêts đã đề cập, Health Check sẽ kiểm tra các Instance, một instance nào đó lỗi thì sẽ không redirect request đến nó nữa

  • Bạn vào lại EC2 Dashboard, Stop một Instance trong Target Group
  • Quay trở lại Targer Group để kiểm tra và đây là kết quả, Health Check bảo unused
  • Như vậy request sẽ không được chuyển tới Instance này nữa, mở trình duyệt và kiểm tra domain của ALB, kết quả là IP Instance không đổi

Bước 5: Khôi phục lại Instance

  • Bạn vào lại EC2 Dashboard, Start lại Instance đã Stop trong Bước 4
  • Test lại domain ALB và đã thấy nó thay đổi IP qua các Instance =))

Lab 02: Auto Scaling Groups Hands On

Bước 1: Tạo Auto Scaling Groups

Từ Sidebar bên trái => AUTO SCALING => Auto Scaling Groups => Create Auto Scaling Group

Step 1: Choose launch template or configuration

  • Name: DemoASG
  • Launch template => Create a launch template

Then you need to create a launch template. So for this, we need to create how and define how the EC2 instances are going to be created.

Create launch template

Khoan tiếp tục các bước tạo Auto Scaling Group, chúng ta cần tạo teamplate trước

  • Launch template name: MyTemplate
  • Phần Launch template contents
    • AMI: Amazon Linux 2
    • Instance type: t2.micro (free tier)
    • Key pair: EC2 Tutorial
    • Network settings: VPC
    • Security group: Tự chọn
  • Các mục khác nếu cần custom thì bạn tự config, không thì bỏ qua

  • Advanced details

    • Tìm User data: Đoạn code ở bài lab trên

  • Click Create a template

Trở lại phần tạo ACG

  • Launch template: Chọn template vừa tạo (MyTemplate)
  • Review lại các thông số ngay dưới phần chọn template => Click Next

Step 2 : Configure settings

  • Purchase options and instance type: Adhere to launch template
  • Subnets: Chọn hết (đảm bảo tính khả dụng cho nhiều AZ)
  • Click Next

Step 3: Specify load balancing and health checks

  • Load balancing: Enable load balancing. Chọn target group đã tạo ở bài lab trên
  • Health checks: Check thêm ELB
  • Click Next

Step 4: Configure group size and scaling policies

For the group size, this is where you can define how it scales.

  • Desired capacity: 2
  • Minimum capacity: 1

  • Maximum capacity: 4

  • Scaling policies: Target tracking scaling policy => tôi muốn tự động. Sau đó bạn điền các thông số ngưỡng phía dưới. Tuy nhiên trong phạm vi bài thực hành, chúng ta sẽ không kiểm thử được các phần tự động nên sẽ chọn None

Bước này định nghĩa khi nào chúng ta sẽ tự động scale

Tuy nhiên trong phạm vi bài thực hành, chúng ta sẽ chọn None để scaling thủ công xem chúng hoạt động ra sao.

  • Click Next

Step 5: Add notifications

(Tạm bỏ qua) - Click Next

Review and Create

Xem lại các cấu hình của bạn, mặc định sẽ có 2 instance trong group, xem trong tab Instance managament nhé :D

Và khi vào EC2 dashboard, bạn sẽ thấy 2 instance được khởi tạo và đang chạy

Check thêm trong target group, bạn cũng thấy chúng đã được tự động register vào đây

Bạn có thể test lại domain của Load Balancing để xem nó có work không nhé =))

Bước 2: Kiểm chứng thêm sức mạnh của Auto Scaling Group

  • Vào dashboard EC2, terminate một Instance trong target group
  • Do đã config desired capacity là 2, vậy điều gì sẽ xảy ra khi 1 instance bị hỏng? Nó sẽ tự tạo instance thay thế

But after the instance is terminated, the actual capacity of my auto scaling group is going to be 1.And therefore, what will happen is that, when the auto scaling group will detect the fact that my previous EC2 instance has been terminated, it will go ahead and create a new instance to match the desired capacity.

You can notice this if you go to activity notification, so I’ll just pause the video to show you.

Chuyển tab Activity để xem AWS đã làm gì nhé

Thấy chi tiết các activity luôn nè

Bạn thấy rõ sự thay thế Instance lỗi trong AWS =)). Trở lại EC2 Instance Dashboard để xem thêm

Instance management

Tất cả được làm một cách tự động và trong suốt với người dùng. Tuyệt vời!

Bước 3: Hãy thay đổi cấu hình

  • Thay đổi Desired capacity thành 4 xem lại các config của targer group, EC2 Dashboard, Auto Scaling Group (Activity, Instance management) xem chuyện gì xảy ra)

Happy Coding!
