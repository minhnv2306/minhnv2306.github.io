[Lab] Section 6: EC2 Instance Storage

Cùng xem các bài Lab phần section 06 nhé

Lab 01: EBS Hands On

Bước 1: Lauch một EC2 Instance

Phần này các bạn tự thực hiện nhé, OS bất kì

Bước 2: Xem Volume

Chọn Instance => Storage => Block devices

Để ý cột Root device type đang là EBS

Chọn chi tiết Volume này, xem chi tiết.

Bước 3: Tạo Volumne chơi chơi

Vào EC2 => Elastic Block Store => Volumes => Create Volume

  • Volume Type: chọn gp2
  • Size: 2GB (để test thôi)
  • Availability Zone*: cái này quan trọng, bạn cần chọn đúng Availability Zone của EC2 để có thể gán (theo đúng lý thuyết) (vào màn EC2 để xem Availability Zone
  • Bỏ qua cột Snapshot ID và Encryption
  • Click Create Volume

Kiểm tra màn Volume, đã có 1 volume mới được tạo với state là available

Bước 4: Attach đến Instance chơi chơi :D

Chọn chuột phải vào Volume => Attach Volume. Sau đó chọn Instance bạn đã tạo từ trước

  • Instance: chọn Instance trước đó bạn đã tạo
  • *Device: để mặc định
  • Click Attach. Chờ 1 khoảng thời gian để attach thành công!

Quay trở về Instance để xem lại khối Block devices

Làm sao để sử dụng Volume => Search “formats EBS volume attach on EC2 instance”

Bước 5: Thực hành kiểm tra lý thuyết

  • Tạo một volume ở Availability Zone khác để không attach được

  • Delete instance, chuyện gì sẽ xảy ra

Về lý thuyết, các EBS sẽ vẫn còn ngay cả khi instance bị terminate, kiểm chứng nhé =))

Tất nhiên chỉ còn Volume mình tạo, Volume đi cùng EC2 vẫn mất

Lab 02: EBS Snapshot Hands On

Như phần lý thuyết đã đề cập, các EBS chỉ có thể sử dụng attach cho các Instance cùng AZ. Bài thực hành sẽ tạo snapshot cho EBS để có thể dùng ở các AZ như hình mô tả

Từ EBS ở bài Lab 01

Bước 1: Tạo snapshot

Chọn EBC => Create Snapshot

  • Description: Demo Snapshot
  • Create Snapshot

Xem snapshot đã được tạo trong EC2 => Elastic Block Store => Snapshots

Như lý thuyết đề cập, snapshots này sẽ không link đến một AZ cụ thể nào, do đó ta có thể gán mọi AZ

Bước 2: Copy sang AZ khác

Chọn Snapshots => Action => Copy sang AZ khác, bất cứ region nào bạn muốn

Bước 3: Tạo Volume

Từ snapshot bạn có thể tạo Volume với thay đổi type, size

Chọn Snapshots => Action => Create Volume

  • Volume Type: Bạn có thể chọn theo mong muốn
  • Size: Bạn có thể chọn theo mong muốn
  • Availability Zone: Bạn có thể chọn theo mong muốn

Bằng cách này bạn có thể sao chép volume sang AZ khác nhau để attach vào các EC2 instance khác AZ

Lab 03: AMI Hands On

Bước 1: Lunch 1 EC2 instance Amazon Linux 2

User data như trong code, có index.html

  • EC2 cài đặt httpd
  • Có index.html
  • Chọn SG cho phép mở cổng 80

Sau đó chờ 1 lúc để EC2 khởi động, bootstrap, sau đó cần 1 khoảng thời gian để chạy EC2 data script. Truy cập IP v4 để xem kết quả

We’re getting a connection error but if I refresh this page, for example, and wait a little bit, at some point the EC2 data script will be executed and the httpd web server will be created.

So it’s always good to demonstrate that not everything is immediate. And this is also one of the reason we create an AMI. So all this bootstrap time, which is going to be 20, 30 seconds for this EC2 instance, will be gone once we create an AMI.

Bước 2: Tạo AMI

Click chuột phải vào Instance => Images and templates => Create Image

  • Image name: DemoImage

Click Create Image

AMI đã được tạo trong Images => AMIs và giờ đây bạn hoàn toàn có thẻ lauch EC2 Instance mới từ AMI này

Bước 3: Lauch Instance mới với AMI

Chọn Lauch Instance. Ngay Step 1 chọn My AMIs (chú ý các AMIs cần trạng thái available để có thẻ hiển thị ở đây)

  • Tiếp tục các bước tiếp theo Add Tags có thể thêm Name = From AMIs

Bước 4: Kiểm tra

Truy cập IP từ Instance, được kết quả giống với Instace đã tạo trước đó

Trong trường hợp EC2 Instance được build từ AMI cũng có EC2 data, EC2 này sẽ chạy sau EC2 data từ AMI, bạn có thể custom lại file index.html để thử nhé :D

Lab 04: EC2 Image Builder Hands On

Bài Lab 03 hướng dẫn bạn tạo AMI một cách thủ công, tạo từ EC2 mà bạn launch trước đó. Tuy nhiên bạn hoàn toàn có thể tạo AMI một cách tự động và dựa vào các image có sẵn của AWS (ví dụ như tạo instance có sẵn aws cli, hay java) qua EC2 Image Builder Service - một dịch vụ của AWS.

Phần thực hành này chúng ta sẽ sử dụng EC2 Image Builder để build các AMI một cách tự động hoặc thủ công (trong bài thực hành sẽ không dùng schedule nên sẽ là thủ công) với nhiều phần mềm, package được tích hợp sẵn cho Instance.

Let’s start!

Bước 1: Tạo Pipeline

Chọn EC2 Image Builder Service => Create pipeline

Step 1

  • Pipeline name: MyDemoPipeLine
  • Build Schedule: Manual
  • Next step 2

Step 2: Chọn recipe

  • Create new recipe
  • Chọn AMI
  • Name: MyDemoRecipe
  • Version: 1.0.0
  • Select image: Select managed images => Amazon Linux 2
  • Image origin: Quick start (Amazon-managed)
  • Image name: Amazon Linux 2 x86
  • Auto-versioning options: Use latest available OS version
  • Trong mục Components: Mục này chọn component theo mục đích của bạn, ví dụ Amazon-corretto-11-headless có Java 11 được cài sẵn. Bạn cũng có thể tự tạo component cho riêng mình. Tạm chọn aws-cli-version-2-linux
  • Chọn test component: Tạm skip
  • Click Next

Step 3: Tuỳ chỉnh cấu hình

  • Chọn “Create a new infrastructure configuration”
  • Name: MyDemoInfra
  • IAM: chọn IAM tương ứng (EC2InstanceProfileForImageBuilder): Các Role Policy cần thiết
  • Tiếp là chọn AWS Infrastructure
  • Chọn Instance type: t2.micro
  • Click Next

Step 4: Distribution settings

  • Chọn “Create distribution settings use service defaults” hoặc cũng có thể tạo “Distribution Settings” cho riêng bạn nếu bạn muốn tạo AMI cho nhiều region. Trong bài thực hành, ta tạm chọn defaults
  • Chọn Next

Step 5: Review and Create pipeline

Bước 2: Run Pipeline

Sau khi pipeline được tạo. Chọn Pipeline đó => Action => Run pipeline. Nhấn vào version để xem chi tiết

Vào EC2 để kiểm tra EC2 mới được build.

Trong EC2 đó bạn có thể xem thêm:

  • Chọn Tags để xem chi tiết hơn
  • Kiểm tra IAM

Xem trạng thái Pipeline. Từ building chuyển sang testing. Khi trạng thái testing, chuyển sang EC2 Dashboard và bạn sẽ thấy EC2 trước đó đã bị terminate

  • Xem lại IAM và Tags của Instance Testing

Check lại status pipeline sẽ chuyển từ testing sang distributing

Lại kiểm tra lại EC2 Dashboard

Cuối cùng chuyển về trạng thái available => Done

And so a recipe is a document that defines how the source image is going to be customized.

So components, is how do we want to customize our image?

Bước 3: Kiểm tra

Sau khi bước 2, chúng ta đã có AMI cài cli và java theo như đã chọn ở bước 1. Bước này ta sẽ build EC2 instance từ AMI đó để kiểm chứng.

Lauch Instance => MyAMIs => Chọn AMIs tạo từ pipeline. Các bước còn lại bạn tự chọn

Sau khi lauch xong EC2 hãy connect ssh vào instance và kiểm tra thành quả

OK rồi nhé :D

Bước 4: Clean up

  • EC2 Dashboard => Terminate EC2 Instance vừa tạo
  • Images => AMIs => xoá AMI
  • Elastic Block Store => snapshots => xoá snapshot

Không cần xoá EC2 Image Builder vì nó không làm gì đâu =))

Good luck to you <3!
